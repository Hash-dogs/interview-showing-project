# 简历上传
## 一、上传整体步骤：
```angular2html
前端上传文件
↓
① 基础合法性校验（最先做，最便宜）
- 文件大小是否超限、文件后缀是否允许
- 直接拒绝明显不合法的，不浪费后续资源
↓
② SHA-256 哈希计算 + 去重检查
- 计算哈希值，查 DB 是否存在
- 已存在 → 直接返回已有的 fileKey/文档ID，流程结束
↓
③ Tika 基础解析（轻量/中度耗时）
- 提取元数据（作者、页数、创建时间等）
- 验证文件真实类型（防止后缀造假）
- 提取基础文本内容
↓
④ 文本判空与 OCR 兜底（⚠️ 新增核心分支）
│
├─ [文本为空/纯乱码] (说明是扫描件/图片PDF)
│       ↓
│    调用 OCR 服务 (带版面分析)
│       ↓
│    获取文本 + 版面信息 (Bounding Box, 块类型)
│       ↓
│    标识来源 `source_type = OCR`
│
└─ [文本非空] (原生PDF/Word等)
↓
标识来源 `source_type = TIKA`
↓
⑤ 统一文本清洗 (TextCleaningService.cleanText)
- 去掉不可见字符、多余换行、规范格式
- 注意：如果是 OCR 提取的文本，清洗时尽量保留原有的版面换行线索
↓
⑥ 文档分块策略（🤖 核心 RAG 逻辑）：默认OCR后文本也是有结构文本
结构质量评估 (Structure Evaluation)
│
├─ 极短文本 (如<300字) ──→ 整体作为一个 Chunk，不分块
├─ 有标题层级 (正则/Markdown标记特征) ──→ 结构感知 + 父子分块 (主路径)
└─ 无结构线索 (大段纯文本) ──→ 语义分块（按句子划分，句与句之间进行向量相似度分析，骤降的时候划分）
↓
⑦ 向量化与多级存储（⚠️ 新增 RAG 存储逻辑）
- A. RustFS：上传文件本体 → 拿到 fileKey
- B. 关系型数据库 (MySQL/PostgreSQL)：存入文件元信息、fileKey、fileHash
- C. 向量数据库 (Vector DB - 如 Milvus, PgVector)：
- 调用 Spring AI 的 EmbeddingModel 将分好的 Chunks 转化为向量 (Vectors)
- 将 Chunk文本、向量、关联的 fileKey 以及 父子ID关系 存入向量数据库
↓
⑧ 返回处理结果给前端
- 返回状态、fileKey、分块数量等
```



# 简历分析

## 一、 从各类文件中提取文本 Apache Tika
1. 一个非常成熟的开源"内容分析工具箱"，它最大的特点是能从上千种不同格式的文件 （如PDF、Office文档、音频、视频等）中，通过统一接口提取出文本和元数据（Metadata）
2. 主要功能是做文本转换，对于文本的噪声处理功能十分有限，只有下面提及几个：
    * **能识别文件类型**（"application/pdf" "image/png"），格式和HTTP请求中的类型描述几乎一致
    * 能根据字符等单位截断文本（如只获取前10000个字符文本）
    * 语言检测（十分不稳定）


## 二、 简历简单除噪：
* TextCleaningService.cleanText()：使用正则表达式对TIka转化后的噪音进行除噪。

2. 非法转义字符解析校准
* 本质：大模型可能会不加转义字符直接输出一些非法句子："他说"你好"然后离开了"。这里的双引号没转义就会被认为是字段的结束符号二错义
* 使用有限状态机思想解决：两个bool类型字段inString表示是否进入json字段，escape字段表示当前字符是否会被转义。
```
    private String repairUnescapedQuotesInJsonStrings(String content) {
        if (content == null || content.isBlank()) {//判断是否为null串以及空白字符串
            return content;
        }
        StringBuilder repaired = new StringBuilder(content.length() + 16);//由于要加如\作转义字符所以会比原串多
        boolean inString = false;
        boolean escaping = false;
        for (int i = 0; i < content.length(); i++) {//逐个字符扫描处理
            char ch = content.charAt(i);
            //当不是inString的时候遇到“说明是起始的“
            if (!inString) {
                if (ch == '"') {
                    inString = true;
                }
                repaired.append(ch);
                continue;
            }
            //说明上一个字符是转义字符
            if (escaping) {
                repaired.append(ch);
                escaping = false;
                continue;
            }
            //原串遇到\转义字符的时候下一个字符直接输出就好
            if (ch == '\\') {
                repaired.append(ch);
                escaping = true;
                continue;
            }
            if (ch == '"') {
            //算法关键所在：判断下一个字符是否为终结符（，] } ;）出错点常在这，判断结束符号不正确导致改错失败
                if (isLikelyJsonStringTerminator(content, i + 1)) {
                    inString = false;
                    repaired.append(ch);
                } else {
                    //append函数中也需要转移，实际添加的是\\和\",到了json原串中就是\"
                    repaired.append("\\\"");
                }
                continue;
            }
            repaired.append(ch);
        }
        return repaired.toString();
    }
```
